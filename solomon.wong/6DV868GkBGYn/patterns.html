Three patterns that do the heavy lifting

Publish/subscribe moves facts around. Event sourcing makes facts the system of record. CQRS lets the write side and the read side stop compromising with each other. They compose — but each earns its keep separately.

Pattern 01

Publish/subscribe

The foundational pattern. A producer publishes an event to a named topic; the broker delivers a copy to every subscriber group that has registered an interest. Within a group, instances share the work; across groups, everyone sees everything. The producer's code never changes when subscribers come and go.

Watch out for: topics quietly becoming APIs. The event schema is a public contract — version it (order-events.v2), evolve it additively, and never assume you know all the consumers.

Pattern 02

Event sourcing

Most systems store current state and throw the history away. Event sourcing inverts that: the append-only log of events is the system of record, and current state is derived by replaying the log from the start (folding each event into the state). You never lose the “why” behind a balance — it is the balance.

Watch out for: schema evolution. Events written five years ago must still be readable today, so upcasting old versions is a permanent responsibility — and deleting personal data from an immutable log needs a strategy (crypto-shredding is the usual one) designed in from day one.

Pattern 03

CQRS — Command Query Responsibility Segregation

One data model rarely suits both sides of an application. Writes want normalised structure and strict invariants; reads want flat, denormalised views shaped exactly like the screen. CQRS splits them: a write model validates commands and records events, and one or more read models are projected from those events, each optimised for a particular query.

Watch out for: applying CQRS everywhere. It shines where read and write shapes genuinely diverge or read traffic dwarfs writes; for a straightforward form-over-table feature, one model and one database remain the right answer.

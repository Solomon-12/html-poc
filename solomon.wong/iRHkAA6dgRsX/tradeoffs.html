Delivery guarantees, and an honest comparison

Networks drop packets, brokers restart and consumers crash mid-message. Every messaging system must decide what happens then — and each answer trades data loss against duplicates against complexity.

01 — Delivery semantics

At-most-once, at-least-once, exactly-once

The three guarantees differ in one question: what does the sender do when it isn't sure the message arrived? Give up (at-most-once), retry (at-least-once), or retry and make duplicates harmless (exactly-once processing).

At-most-once

Cheap, lossy

Lowest latency and overhead; acceptable when a lost message costs nothing — metrics ticks, presence pings, sensor readings that will be superseded a second later.

At-least-once

The practical default

What Kafka, SQS and RabbitMQ give you out of the box. Pair it with idempotent consumers — keyed on an event ID — and duplicates become a non-event.

Exactly-once

Expensive, bounded

Kafka transactions achieve it within a Kafka-to-Kafka pipeline; the moment you touch an external system (an email, a payment API), you are back to designing idempotency yourself.

02 — The honest comparison

Events versus request/response

Neither style wins outright. Request/response keeps its considerable virtues — immediacy, simplicity, easy debugging — and mature systems use both deliberately. Here is the scorecard without the marketing.

Event-driven versus request/response, dimension by dimension
Dimension Event-driven Request/response
Coupling Loose — producers know topics, not consumers; new consumers need no producer change Tight — caller knows the callee's address, API shape and availability
Getting an answer Awkward — replies need correlation IDs and a return topic, or a separate query Natural — the answer is the response, in the same interaction
Consistency Eventual — consumers lag the truth by milliseconds to minutes Immediate — the caller sees the result before proceeding
Failure isolation Strong — a dead consumer just builds a backlog; others are untouched Weak — a slow callee stalls the caller; failures cascade without circuit breakers
Load spikes Absorbed — the broker buffers and consumers drain at their own rate Punishing — spikes become timeouts and retries at the worst moment
Debugging & tracing Harder — causality spans asynchronous hops; needs correlation IDs and good tooling Easier — one stack trace, one timeline, one place to look
Operational surface Larger — broker, schema registry, consumer lag, dead-letter queues, replay tooling Smaller — a load balancer and the service itself
Evolving contracts Additive schema changes are easy; breaking changes need versioned topics API versioning is well-trodden; breaking changes still hurt, but visibly
Replay & audit Built in — the log is history; rebuild views or backfill consumers from it Absent — unless you add an audit log as a separate concern
Best fit Fan-out reactions, integration across teams, spiky pipelines, audit trails Lookups, validations, anything where the caller must know the outcome now

The honest summary: events buy you decoupling and resilience with the currency of eventual consistency and operational complexity. If you are not prepared to spend that currency — idempotent consumers, monitoring for consumer lag, a dead-letter strategy — request/response is not a lesser choice; it is the correct one.

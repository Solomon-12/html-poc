GitFlow

Vincent Driessen's 2010 model gives every kind of change its own lane: two permanent branches (main and develop) and three supporting branch types (feature, release and hotfix) with strict rules about where each may start and where it must merge.

One release cycle, drawn out

The rules of the model

  • main holds production history only. Every merge to it is a release and gets a version tag.
  • develop is the integration branch: the state of the next release, always ahead of main.
  • feature/* branches start from develop and merge back into develop. They never touch main.
  • release/* branches start from develop when a release is feature-complete. Only stabilisation fixes land here; new work continues on develop in parallel. When signed off, the branch merges into main (tagged) and back into develop.
  • hotfix/* branches start from main to patch production urgently, then merge into both main and develop.

When GitFlow still makes sense

GitFlow has fallen out of fashion for continuously deployed web services — even Driessen added a note in 2020 saying as much — but the model was never designed for them. It remains a sound choice when releases are genuinely discrete events:

  • Versioned, installed software — desktop applications, SDKs, firmware, on-prem servers — where v1 and v2 are both alive in the field and both need fixes.
  • Store-gated mobile apps, where a release train waits on review and a stabilisation window is unavoidable.
  • Formal sign-off cultures: the release branch gives QA, security review or regulatory approval a frozen target while development continues elsewhere.
  • Long stabilisation phases, because "harden the release" and "build the next one" are cleanly separated lines of history.

Pros and cons

Strengths

  • main is a clean, auditable record: every commit is a tagged, shipped release.
  • Release stabilisation happens on its own branch without freezing everyone else.
  • The hotfix path is explicit — production can be patched without shipping half-done work from develop.
  • Several release lines can be supported in parallel.

Trade-offs

  • Two permanent branches drift apart; integration pain is deferred and compounds.
  • Every release involves ceremony: cut, stabilise, merge twice, tag. Small teams feel the overhead immediately.
  • Changes reach production slowly — the antithesis of continuous delivery.
  • The double merges (release and hotfix back into develop) are easy to forget, and forgetting one silently regresses the next release.

If you are choosing today

Ask the question from the overview flowchart: do you support several release lines at once, or need a frozen target for sign-off? If yes, GitFlow (or a trimmed variant without develop) earns its complexity. If you ship one live version of a web service, trunk-based development will get changes to production faster with far less merge traffic.

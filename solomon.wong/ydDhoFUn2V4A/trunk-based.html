Trunk-based development

One shared branch — the trunk, usually called main — that everyone integrates into at least daily. Branches still exist, but they live for hours or a day or two, not weeks, and the trunk is kept releasable at all times.

What the history looks like

Feature flags do the heavy lifting

Short-lived branches only work if half-finished features can merge safely. Feature flags make that possible by decoupling deployment (the code is on the servers) from release (users can see it):

  • Unfinished work merges behind a flag that is off in production, so integration never waits for feature completeness.
  • Release becomes a configuration change — flip the flag for 1% of users, watch the dashboards, ramp up — and rollback is flipping it off, not reverting a merge.
  • Flags are debt: every flag needs an owner and a removal date. A codebase full of stale flags is harder to reason about than the long-lived branches it replaced.

What your CI must be able to do

Trunk-based development moves the safety net from branching structure to automation. Before adopting it, be honest about whether these hold:

  • Fast pre-merge checks. Build plus unit tests in under ten minutes on every proposed commit; anything slower moves to a post-merge stage.
  • A merge queue. Commits are validated against the exact state of main they will land on, so two individually green changes cannot break the trunk together.
  • Stop-the-line discipline. A red main is everyone's problem. The default response is revert first, investigate later — the trunk must return to green in minutes, not hours.
  • Post-merge depth. Slower integration and end-to-end suites run continuously against the trunk, and their failures page the author of the merge that broke them.
  • Deployment from main. Releases are cut (and tagged) straight from the trunk — if deploying main right now feels frightening, fix that before deleting your branches.

How releases are cut

Trunk-based teams release in one of two ways, and the choice is about audit needs rather than safety:

  1. Release from trunk. Every deploy is a tagged commit on main — the pattern in the diagram above. Rollback means redeploying the previous tag. This is the default for continuously deployed services.
  2. Branch for release. A branch such as release/3.2 is cut from the trunk at the release point and receives only cherry-picked fixes — never direct feature work, and it is never merged back. Teams shipping versioned artefacts (mobile builds, installers) use this without giving up daily integration on the trunk.

Note what is absent in both cases: there is no develop branch, no long stabilisation phase, and fixes always land on the trunk first, then flow outwards to any release branch that needs them.

Pros and cons

Strengths

  • Integration pain is paid in small daily instalments instead of a monstrous merge at release time.
  • Small diffs get better reviews — a 200-line change is reviewable, a 5,000-line branch is skimmed.
  • The deployable artefact is always known: it is main.
  • One long-lived branch keeps CI simple and cheap.
  • It is the natural substrate for continuous delivery and progressive rollout.

Trade-offs

  • Weak test automation is fatal — the trunk is only as safe as the checks guarding it.
  • Feature-flag debt accumulates unless removal is actively managed.
  • Slow review cultures turn "short-lived" branches into long-lived ones with a different name.
  • Formal release sign-off or several supported versions in production push you back towards release branches — see GitFlow.

When to choose it

Trunk-based development fits teams shipping a single live version of a service — web applications, APIs, internal platforms — that deploy at least daily and already trust their CI. If you ship versioned software on a schedule, or must keep old release lines alive, read the GitFlow page instead: the answer there is different for good reasons.
